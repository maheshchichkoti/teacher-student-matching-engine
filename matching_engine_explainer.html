1 What does it do?
Before this: ops team manually searched for a teacher when a new student signed up — took ~10 minutes per student.

Now: call one API with a student_id + their preferred schedule → get back top 3 teachers ranked by match score in ~11 seconds.

Each result includes the teacher's match score, trial conversion rate, lesson quality score, and their available time slots that match the student's schedule.
2 How matching works — 3 steps
Step 1 — Hard Filter

Remove ineligible teachers

Only keep teachers where status = 'active' and language = 'EN' (teaching English). Teachers with HE or AR as teaching language are excluded. This is the gate — if a teacher doesn't pass, they never get scored.

↓
Step 2 — Availability Check

Find teachers with matching slots

Check each teacher's weekly recurring slots (from teacher_availability table — stored as JSON per day). Filter slots that fall within the student's preferred days and time range. Teachers with zero matching slots are dropped.

↓
Step 3 — Scoring

Rank remaining teachers by match score

Each teacher gets a weighted score from 5 components (see Section 4). Sort descending. Return top 3.

↓
Output

Top 3 teachers + available slots

Each result shows: name, match score %, trial conversion rate, lesson quality score, available slots, current students, free capacity.

4 Scoring — how the match % is calculated
Weights match the spec exactly. Each component is normalized to 0–1 before weighting.
Student Fit 30%
Language match (70%) + level match (30%). Level is neutral (0.5) until data is available. Language fit = 1.0 for all EN teachers.
Availability Fit 25%
Number of matching slots ÷ 5. 5+ matching slots = 100%. Fewer slots = lower score.
Teacher Performance 20%
Trial conversion rate (60%) + lesson quality score (40%). Both from real DB data. Conversion from trial_class_registrations. Quality from lesson_feedbacks.
Recurring Compatibility 15%
How many of the student's preferred days are covered. 2 days requested + teacher has both = 100%.
Teacher Capacity 10%
Free slots = MAX_CAPACITY (20) − current_students. Teachers near full load get a lower score. Distributes students evenly.
5 DB queries — all batched, 5 total per request
Performance optimization: instead of querying per-teacher (97 individual queries), all data is fetched in 5 batch queries before the scoring loop. This brought response time from ~113s to 11s.
#QueryPurposeResult
1 SELECT id, full_name, language FROM users WHERE role_name='teacher' AND status='active' Hard filter — active EN teachers 82 eligible teachers
2 SELECT teacher_id, SUM(status='converted')/COUNT(*) FROM trial_class_registrations GROUP BY teacher_id Real conversion rates for all teachers Map: teacher_id → conv%
3 SELECT teacher_id, AVG((grammar+pronunciation+speaking)/3) FROM lesson_feedbacks GROUP BY teacher_id Real lesson quality for all teachers Map: teacher_id → quality%
4 SELECT teacher_id, COUNT(DISTINCT student_id) FROM classes WHERE status IN (...) GROUP BY teacher_id Current student load per teacher Map: teacher_id → count
5 SELECT user_id, mon,tue,wed,thu,fri,sat,sun FROM teacher_availability WHERE user_id IN (...) Weekly availability for all eligible teachers in one shot Map: teacher_id → slots JSON
6 API — how to call it
POST http://localhost:5000/match 
Request body:
{
  "student_id": 930,
  "preferred_days": ["Sunday", "Tuesday"],
  "preferred_time_from": "16:00",
  "preferred_time_to": "21:00",
  "mode": "trial"
}

student_id     → fetches real student data from DB (name, language, learning goal)
preferred_days → days agent tells us student can do (not in DB yet)
preferred_time → time range agent tells us (not in DB yet)
mode           → "trial" or "subscription"

8 Assumptions & limitations
Only 2 things are hardcoded per Mahesh's confirmation — student preferred schedule and student goal at signup. Everything else is real DB data.
  • preferred_days / preferred_time HARDCODED — passed in request body. Not captured at student signup yet. Mobile team needs to add these questions at onboarding.
  • Student learning goal (new students) HARDCODED — defaults to "improve_conversation" for students with no prior classes. For returning students, real goal is fetched from classes.student_goal.
  • MAX_CAPACITY = 20 HARDCODED — teacher max capacity not in DB. Will be updated once teacher form data is imported.
  • trial_conversion_rate REAL — computed live from trial_class_registrations (status = 'converted' / total). The users.trial_conversion_rate column was mostly 0 so not used.
  • lesson_quality_score REAL — average of grammar_rate + pronunciation_rate + speaking_rate from lesson_feedbacks (101,571 rows). Normalized to 0–100.
  • Language hard filter — teachers with language = 'HE' or 'AR' are excluded. Only EN teachers shown (Tulkka is an English learning platform). Teachers with NULL language are included (assumed EN).
  • Teacher tags / teaching style SKIPPED — not in DB. Will be added after teacher form responses are imported to DB.
  • Teacher native language matching SKIPPED — native_language column exists in users but 0 rows populated. Will enable once teacher form data imported.
  • Student level matching SKIPPED — only 1,354 / 6,466 students have student_level populated. Too sparse to use in scoring. Level fit set to 0.5 (neutral) for all.
  • current_students count — counted from classes with status in (pending, started, ended). May slightly overcount if old ended classes are included. Can add date filter later.
  • One-time availability (trial slots) SKIPPED — spec mentions a separate one-time calendar from cancellations. Not implemented. Using recurring calendar for both trial and subscription mode in demo.
  • Returns top 3 — Vinay said "top 2 or top 3 at most". Change results[:3] to results[:5] in matching_engine.py to return up to 5.
