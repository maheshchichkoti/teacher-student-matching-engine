1 What does it do?
The matching engine is a standalone FastAPI service on top of the live PostgreSQL warehouse.

You call one API with a student_id (or explicit profile) + their preferred schedule → it returns a ranked shortlist of teachers with scores, availability and capacity that can be booked safely.

Each result includes the teacher's match score, trial conversion rate, retention rate, lesson quality score, their trial slots (for trials), recurring slots (for subscriptions), and basic capacity information.
2 How matching works — 3 steps
Step 1 — Hard Filter

Remove ineligible teachers

Only keep teachers where status = 'active' in clean.teachers, who can teach the requested target_language, and (optionally) satisfy the student's native-language requirement via languages_spoken. If the student requires a native-language teacher and no teacher speaks that native language, they never reach scoring.

↓
Step 2 — Availability Check

Find teachers with matching slots

Use normalized availability from clean.teacher_availability (day_of_week, start_time, end_time) plus the AvailabilityService to:
• Expand each day into 30-minute slots with a canonical 0=Sunday mapping shared with the matching engine.
• For trial mode, call get_trial_availability() to apply the full 3-layer Node.js logic (configured schedule, teacher holidays, existing classes) and keep only slots inside the student's preferred window.
• For subscription mode, call get_recurring_availability() to check a 4‑week pattern for conflicts and build at least one recurring option per preferred day.
Teachers with no viable trial slots (in trial mode) or no recurring slots (in subscription mode) are dropped.

↓
Step 3 — Scoring

Rank remaining teachers by match score

Each teacher gets a weighted score from 5 components (see Section 4). The engine then applies small adjustments for load balancing, teacher priority and age/level-specific personalisation, sorts descending, and returns the top candidates (up to 5 in the current implementation).

↓
Output

Top 3 teachers + available slots

Each result shows: name, match score %, trial conversion rate, lesson quality score, available slots, current students, free capacity.

4 Scoring — how the match % is calculated
Weights match the spec exactly. Each component is normalized to 0–1 before weighting.
Student Fit 30%
Student fit uses age range, CEFR level, tag overlap and learning goals. It is intentionally tag-based and avoids sparse style columns.
Availability Fit 25%
Based on how many concrete slots fall inside the student's preferred window. 5+ matching slots saturate this component; fewer slots reduce it proportionally.
Teacher Performance 20%
Blends three signals from the warehouse: trial conversion rate, 90‑day retention rate and average lesson quality, all normalised to 0–1 and weighted (conversion 40%, retention 30%, quality 30%).
Recurring Compatibility 15%
How many of the student's preferred days have at least one viable recurring slot. If all requested days have a recurring option, this component is 100%.
Teacher Capacity 10%
Free capacity is calculated from max_students_capacity and current active students. Teachers near full load are gently down‑weighted, and under‑utilised teachers get a small boost to balance distribution.
5 DB queries — batched per request
The FastAPI endpoint opens a single PostgreSQL connection per request and fetches all required data in batch queries before entering the scoring loop. No per‑teacher round‑trips are made inside the core loop.

}

preferred_days → schedule constraint supplied by UI / sales workflow
preferred_time → time range supplied by UI / sales workflow
mode           → "trial" or "subscription"

8 Assumptions & limitations
Key point: preferred schedule still arrives via the request body. Core student and teacher matching attributes come from PostgreSQL.
  • preferred_days / preferred_time REQUEST-ONLY — passed in the request body via FastAPI. Not yet captured at student signup; mobile/web clients must continue sending this explicitly.
  • Student learning goal REAL — taken from clean.students.learning_goal when present. If empty, the engine simply treats goals as an empty list (neutral for scoring).
  • MAX_CAPACITY fallback DEFAULT — if clean.teachers.max_students_capacity is NULL, the code uses a constant of 20 as a safe default.
  • trial_conversion_rate REAL — computed from clean.classes trial rows joined with analytics.leads.funnel_stage.
  • lesson_quality_score REAL — average of grammar/vocabulary/fluency scores in analytics.class_facts, normalised to 0–100.
  • Retention REAL — sourced from serve.teacher_performance_profile.student_retention_rate.
  • Teacher tags REAL — teacher_tags in clean.teachers drive fit and personalisation adjustments.
  • Teacher native language matching REAL — implemented via clean.teachers.languages_spoken when requires_native_language_teacher = true. If the student's native language is not present in that list, the teacher is filtered out.
  • Student level matching — uses clean.students.cefr_level when present; otherwise that dimension falls back to neutral (0.5) in compute_student_fit.
  • current_students count REAL — calculated from subscription + non‑subscription classes as implemented in fetch_all_student_counts. It may slightly over‑ or under‑count in unusual edge cases but is consistent across all teachers.
  • One-time vs recurring availability — one‑time trial slots use the full 3‑layer availability logic; recurring slots are built off clean.teacher_availability + conflict checks only, as in the original Node.js implementation.
  • Returned shortlist size — the FastAPI endpoint currently returns up to 5 teachers per call (results[:5]), sorted primarily by match score and retention, with tiebreakers on slots and capacity.
